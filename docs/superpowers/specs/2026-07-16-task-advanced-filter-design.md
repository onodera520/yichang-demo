# 任务协同高级筛选设计

## 目标

为 `/tasks` 页面现有的“更多筛选”按钮增加可用交互，帮助用户按任务自身属性进一步缩小列表范围。视觉和操作方式参考订单异常页的高级筛选浮层，但不修改或复用订单业务字段。

本次只改任务协同页的高级筛选区域，不调整订单异常页、公共布局、任务表格字段或其他任务操作。

## 方案比较

### 方案 A：按钮下方浮层（采用）

点击“更多筛选”后，在按钮下方打开右对齐浮层。浮层内编辑草稿条件，点击“应用筛选”后才更新任务列表。

优点：

- 与订单异常页交互一致，学习成本低。
- 不占用任务列表的固定纵向空间。
- 可保留“应用”和“取消”的明确操作边界。

### 方案 B：筛选区展开第二行

点击按钮后在现有筛选区下方展开更多字段。

优点是所有条件始终可见；缺点是会压缩任务表格高度，并改变当前高保真布局，因此不采用。

### 方案 C：右侧抽屉

使用独立抽屉承载筛选条件。

适合大量复杂筛选，但会与任务详情右栏产生视觉和交互冲突，本次字段较少，不采用。

## 筛选字段

高级筛选包含四组条件：

1. 任务标题：文本包含匹配，忽略首尾空格和大小写。
2. 来源对象编号：文本包含匹配，用于订单号、SKU 或其他来源编号。
3. 创建时间：开始日期与结束日期，边界日期包含在结果内。
4. 剩余 SLA：最小小时数与最大小时数，边界值包含在结果内。

现有负责人、异常来源、风险等级和截止时间继续保留在快捷筛选区，不在浮层内重复。

## 交互设计

- 点击“更多筛选”打开或关闭浮层。
- 浮层宽度约 420px，右侧与按钮对齐，视觉风格参考订单高级筛选。
- 打开浮层时，将当前已应用条件复制到草稿状态。
- 修改草稿不会立即改变任务列表。
- 点击“应用筛选”后更新列表、回到第一页并关闭浮层。
- 点击浮层内“重置”只清空高级筛选草稿，不影响顶部快捷筛选。
- 页面现有“重置”按钮清空快捷筛选和已应用的高级筛选。
- 点击浮层外部或按 `Escape` 关闭浮层，不应用未提交的草稿。
- 有已应用条件时，“更多筛选”按钮显示蓝色激活状态和条件数量徽标。
- 条件数量按逻辑条件计算：创建时间范围计 1 项，SLA 范围计 1 项。

## 校验与空状态

- 创建时间开始日期晚于结束日期时显示错误，并禁用“应用筛选”。
- SLA 只接受大于等于 0 的数字。
- SLA 最小值大于最大值时显示错误，并禁用“应用筛选”。
- 没有匹配任务时沿用任务表格现有空状态，不新增页面级提示。

## 数据与状态

- 高级筛选状态只保存在 `Tasks` 页面 React 会话中，不写入 `localStorage`。
- 筛选在现有任务数据上执行，不修改 `mockData.js`。
- Tab、顶部搜索、快捷筛选和高级筛选采用 AND 关系。
- 状态流转提醒定位任务时，继续沿用现有逻辑清空所有筛选；高级筛选也必须被清空，避免定位结果不可见。

## 组件边界

- 新建任务专用浮层组件，负责草稿状态、关闭行为、表单和条件徽标。
- 新建纯函数模块，负责默认值、匹配、校验和条件计数。
- `Tasks.jsx` 只负责维护已应用条件、组合列表过滤以及重置/定位时清空条件。
- 不编辑订单高级筛选组件及其业务规则。

## 测试与验证

- 纯函数测试覆盖文本匹配、日期边界、SLA 边界、AND 组合、校验和条件计数。
- 组件契约测试覆盖浮层打开、点击外部关闭、`Escape` 关闭、应用、重置和徽标。
- 页面交互测试覆盖应用后回到第一页、页面重置清空高级条件，以及任务提醒定位时清空高级条件。
- 运行 `node --test`、`npm.cmd run build` 和 `git diff --check`。
- 在 1440x900 下验证浮层不遮挡任务详情右栏、不造成页面溢出，且只修改任务协同页。
