# 任务负载调度设计

## 目标

将任务协同页现有的“团队任务概览”从静态负载展示升级为可解释、可预览、需人工确认的任务调度闭环。

## 范围

- 仅修改 `/tasks` 页面及其任务业务状态。
- 使用确定性前端规则，不宣称使用真实 AI、排班系统或后端预测模型。
- 所有成员容量、擅长异常类型和可用状态使用 `mockData.js`。
- 调度只改变任务负责人，不改变任务状态、SLA、风险等级和来源异常状态。
- 自动方案必须先预览，用户确认后才能执行。

## 成员模型

每位成员包含：

- `name`：负责人姓名。
- `capacity`：可承载的标准任务权重。
- `expertise`：擅长的异常来源数组。
- `availability`：`可接单`、`忙碌`或`不可用`。

不可用成员不参与推荐。当前负责人不作为自己的接手候选人。

## 负载评分

每个未完成任务的基础权重为 `1`，再叠加：

- 高风险 `+0.35`，中风险 `+0.15`。
- 已超时 `+0.8`。
- 剩余 SLA 不超过 2 小时 `+0.5`。
- 剩余 SLA 在 2 至 8 小时 `+0.25`。

成员负载率为任务权重总和除以成员容量。`85%` 及以上为过载，`65%` 至 `84%` 为繁忙，低于 `65%` 为可接手。进度条最大显示宽度为 `100%`，文字允许显示超过 `100%` 的真实值。

## 推荐规则

接手候选人按以下顺序评分：

1. 转交后的预计负载率越低越优先。
2. 擅长当前任务异常来源的成员获得推荐加分。
3. `忙碌`成员降低优先级，`不可用`成员直接排除。
4. 预计负载超过 `100%` 的候选人不进入前三名。

推荐结果必须展示推荐原因、当前负载和转交后负载，不使用不可解释的“AI 最优”文案。

## 交互

### 团队任务概览

- 保留成员、进行中、已超时、高风险和负载率。
- 过载成员显示红色“过载”标识。
- 顶部增加“生成调度方案”按钮。
- “查看全部成员”打开成员负载详情，不再是无功能按钮。

### 普通转交

- 转交弹窗展示最多三名系统推荐接手人。
- 点击推荐人会选中该负责人。
- 显示原负责人和新负责人的负载变化预览。
- 用户仍可从完整负责人列表中手动选择。

### 调度方案

- 系统从过载成员中选择可转交任务，最多生成五条建议。
- 每条建议展示任务、原负责人、推荐接手人、推荐原因和双方负载变化。
- 用户可以取消单条建议。
- 点击“确认调度”后统一转交剩余建议。
- 如果不存在安全的接手人，显示空状态，不执行任何修改。

### 执行反馈

- 每条被转交任务追加“负载调度”处理记录。
- Toast 显示转交任务数及主要过载成员的负载变化。
- 团队负载条立即重新计算并平滑变化。
- 因任务状态未变化，不产生任务 Tab 数字提醒。

## 安全边界

- 不自动执行调度。
- 不把任务分配给不可用成员。
- 不把预计负载超过 `100%` 的成员作为推荐对象。
- 没有可行方案时不提供“确认调度”。
- 手工转交继续允许用户选择非推荐但可用的成员，并展示负载预警。

## 验证

- 纯函数测试覆盖任务权重、负载率、过载识别、推荐排序、不可用排除和方案生成。
- 页面测试覆盖推荐选择、方案预览、取消单条建议、确认调度和空状态。
- 状态测试覆盖负责人更新、处理记录、负载重算和不产生状态提醒。
- 运行 `node --test` 和 `npm.cmd run build`。
- 在 1440×900 下验证弹窗不溢出、右侧卡片不挤压、负载变化清晰可读。
