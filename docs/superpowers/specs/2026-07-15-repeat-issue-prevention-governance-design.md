# 反复问题防复发治理设计

## 1. 目标

将“数据复盘 > 反复问题识别”从静态分析表格升级为可执行、可追踪、可验证的防复发治理闭环。

系统不仅要回答“哪些问题反复发生”，还必须支持：

- 解释问题为何被判定为反复发生；
- 明确根因、责任人、治理措施和验证标准；
- 将治理动作同步到任务协同；
- 防止同一问题重复创建活动任务；
- 在任务完成后进入观察期，而不是直接宣告问题解决；
- 根据观察结果确认稳定、重新治理或回滚。

首版继续使用 Mock 数据，不接入真实后端。

## 2. 设计原则

1. **治理对象不是单笔异常**：一行代表一个稳定的问题簇，而不是某一张订单。
2. **任务完成不等于问题解决**：执行任务完成后必须经过观察期验证。
3. **同一问题只允许一个活动治理任务**：重复操作必须被拦截并引导到已有任务。
4. **高风险动作必须人工确认**：首版不自动修改库存、物流或平台规则。
5. **效果必须可衡量**：优先使用标准化复发率，不只比较原始出现次数。
6. **操作必须可追踪**：忽略、审批、执行、确认稳定、回滚和重新开启均记录原因与时间。

## 3. 核心数据模型

### 3.1 反复问题 `RepeatedIssue`

每个问题使用稳定的 `issueId`，不能继续将会随时间范围变化的 `source` 文本作为身份。

```js
{
  issueId: 'RI-AMZ-US-STOCKOUT-001',
  source: 'AMZ-US-250601-001',
  platform: 'Amazon',
  store: 'US Store 01',
  issueType: '缺货',
  fingerprint: 'Amazon|US Store 01|缺货|安全库存不足|v2.4',
  status: '待确认',
  riskLevel: '高',
  owner: '未分派',
  firstOccurredAt: '2026-05-26 08:42',
  lastOccurredAt: '2026-06-01 10:24',
  occurrencesByRange: {
    '近7天': 32,
    '近30天': 76,
    '本月': 46,
  },
  exposureByRange: {
    '近7天': 12800,
    '近30天': 49800,
    '本月': 21400,
  },
  amountByRange: {
    '近7天': 1260,
    '近30天': 3420,
    '本月': 3860,
  },
  suggestedAction: '优化补货策略，设置安全预警',
  rootCause: '安全库存阈值未覆盖促销期销量波动',
  failedControl: '补货预警仍使用固定日均销量',
  activeTaskId: null,
  governancePlan: null,
  samples: [],
  auditLogs: [],
}
```

标准化复发率按以下公式展示：

```text
重复事件数 / 业务暴露量 * 1000
```

不同问题的业务暴露量可分别使用订单数、发货票数、SKU-天数或接口调用数。Mock 数据必须明确每个问题采用的口径。

### 3.2 治理方案 `GovernancePlan`

```js
{
  objective: '未来7天同类缺货异常不超过2次',
  action: '调整安全库存并启用促销期动态阈值',
  scope: 'US Store 01 / 关联热销SKU',
  owner: '王敏',
  reviewer: '张晓',
  deadline: '2026-06-03 18:00',
  observationDays: 7,
  targetOccurrences: 2,
  rollbackCondition: '库存周转天数上升超过20%',
  sideEffectMetric: '库存周转天数',
  baselineRate: 2.5,
  observedRate: null,
}
```

### 3.3 关联任务

治理任务沿用现有任务数据结构，同时增加：

```js
{
  sourceKind: 'repeatedIssue',
  sourceId: 'RI-AMZ-US-STOCKOUT-001',
  governanceTask: true,
  verificationCondition: '未来7天同类缺货异常不超过2次',
}
```

任务标题格式固定为：

```text
[防复发] 缺货 - AMZ-US-250601-001
```

## 4. 状态机

完整状态流程：

```text
待确认 -> 诊断中 -> 待审批 -> 治理中 -> 观察中 -> 已稳定
   |          |          |          |          |
   v          v          v          v          v
 已忽略     返回诊断    驳回修改    已回滚    重新治理
                                                |
                                                +-> 治理中
```

状态规则：

- `待确认`：AI识别出的候选反复问题，尚未经过人工确认。
- `诊断中`：已确认需要治理，但根因或方案尚未完成。
- `待审批`：高风险方案等待审核；普通方案可跳过此状态。
- `治理中`：存在活动治理任务，正在执行预防措施。
- `观察中`：治理任务已完成，正在验证复发率和副作用指标。
- `已稳定`：观察期达到目标并由人工确认。
- `已忽略`：判定为误报或当前不需要治理，必须填写原因。
- `已回滚`：治理措施产生副作用或执行失败，必须填写回滚原因。

允许的快捷路径：用户在首次立项时已经填写完整根因和治理方案，可从待确认直接进入待审批或治理中；普通风险方案跳过待审批。高风险方案创建的关联任务在审批前保持不可执行。

禁止行为：

- 无负责人、目标和验证条件时不能开始治理；
- 同一 `issueId` 已有活动治理任务时不能再次创建；
- 没有执行凭证时不能从治理中进入观察中；
- 观察样本不足时不能确认已稳定；
- 任务完成不能直接将问题置为已稳定。

## 5. 页面交互

### 5.1 表格

“反复问题识别”继续保留固定高度、内部滚动和时间范围筛选。

列调整为：

```text
对象来源 | 问题类型 | 出现次数 | 影响金额 | 治理状态
```

现有“建议改进行动”长文本移入详情抽屉，避免截断文本占用表格宽度。

交互要求：

- 整行和对象编号均可打开详情抽屉；
- 当前选中行显示浅蓝色背景；
- 行支持鼠标点击以及键盘 Enter/Space；
- 治理状态使用紧凑状态标签；
- 切换近7天、近30天和本月只改变统计值，不能重置问题状态或任务关联；
- 排序优先考虑风险、标准化复发率和影响金额，而不是只看原始次数。

### 5.2 防复发详情抽屉

复用公共 `DetailDrawer`，宽度设为约 `520px`。主内容区保持可见，抽屉覆盖右侧。

抽屉按以下顺序组织，不使用多层嵌套卡片：

1. **治理摘要**：问题类型、风险、状态、负责人、首次与最近发生时间。
2. **复发证据**：当前时间范围次数、每千业务机会复发率、影响金额、小型趋势、最近三条样本和判定口径。
3. **根因诊断**：确认根因、上次控制措施以及为何再次失效。
4. **防复发方案**：措施、适用范围、负责人、目标、观察窗和回滚条件。
5. **协同与验证**：关联任务、截止时间、执行凭证、观察进度和副作用指标。
6. **操作记录**：确认、修改、审批、执行、回滚和重新开启记录。

抽屉底部主动作根据状态变化：

- 待确认：`启动防复发治理`
- 诊断中：`提交治理方案`
- 待审批：`审批通过`，同时提供次级的 `驳回`
- 治理中：`查看关联任务`
- 观察中：`确认已稳定`，同时提供 `重新治理`
- 已稳定：`重新打开`
- 已忽略或已回滚：`重新治理`

待确认状态提供次级动作 `不是反复问题`，操作时必须选择原因。

### 5.3 创建防复发任务

复用并扩展现有 `TaskCreateModal`，支持 `mode="governance"` 和预填初值。

弹窗文案：

- 标题：`创建防复发任务`
- 说明：`任务将关联反复问题 AMZ-US-250601-001，创建后进入任务协同。`
- 主按钮：`创建并开始治理`

字段：

- 任务标题；
- 负责人；
- 截止时间；
- 根因；
- 防复发措施；
- 生效范围；
- 验证条件；
- 观察周期；
- 回滚条件。

负责人、防复发措施、生效范围和验证条件为必填。高风险措施需要选择审核人，创建后进入待审批。

创建成功后：

- 任务进入 `/tasks`；
- 问题写入 `activeTaskId`；
- 普通方案进入治理中，高风险方案进入待审批；
- Toast 显示 `防复发任务已创建`，并提供进入任务协同的入口。

## 6. 异常与边界处理

### 6.1 重复任务

创建前根据 `issueId` 检查活动任务。若已存在，阻止创建并提示：

```text
该问题已有防复发任务，由王敏处理中
```

主操作改为 `查看关联任务`。

连续点击、重复提交以及切换时间范围后再次操作都不能生成第二条活动任务。

### 6.2 创建失败

- 保留弹窗中的所有输入；
- 问题状态和任务列表均不更新；
- 显示可重试错误；
- 禁止先更新问题状态、后创建任务的部分成功状态。

### 6.3 撤销与回滚

- 创建任务后短时间内可通过 Toast 撤销，恢复问题到创建前状态；
- 任务已经开始后不能直接删除，必须关闭任务并填写原因；
- 治理措施触发回滚条件时进入已回滚，保留原方案、凭证和回滚记录。

### 6.4 再次复发

已稳定问题在后续时间范围内再次超过阈值时：

- 重新打开原治理单；
- 保留上一轮根因、方案、任务和验证记录；
- 状态统一回到待确认，并通知上一轮负责人重新研判；
- 不创建新的独立问题身份。

## 7. 防复发效果指标

北极星指标：

```text
30天治理复发率 = 观察期内再次命中同一问题指纹的治理单数 / 已进入有效观察期的治理单数
```

抽屉和后续复盘需要展示：

- 每千业务机会标准化重复发生率；
- 治理前后同周期复发率变化；
- 平均复发间隔；
- 影响金额变化；
- 观察期通过率；
- 重新开启率；
- 回滚率；
- 规则拦截覆盖率；
- 缺货率、履约时效、退款率等副作用指标。

预计避免损失必须标记为“估算”，并展示计算口径，不能将其作为已实现收益。

## 8. 状态与组件边界

### `mockData.js`

负责提供反复问题、样本、趋势、治理方案和审计日志的初始 Mock 数据。

### `DemoStateContext.jsx`

负责持有跨页面共享的反复问题状态，并提供：

- 打开或忽略治理问题；
- 创建防复发任务并执行去重；
- 审批或驳回治理方案；
- 任务完成后进入观察期；
- 确认稳定、重新治理和回滚；
- 记录审计日志。

所有涉及问题与任务同时变化的操作必须作为一次状态事务完成，避免部分更新。

### `Analytics.jsx`

负责时间范围展示、行选择、打开抽屉和触发治理动作，不自行维护跨页面的治理真相。

### 公共组件

- 复用 `DetailDrawer` 承载治理详情；
- 扩展 `TaskCreateModal` 支持治理模式和预填；
- 复用 `RiskTag`、`PlatformLogo` 和 Toast；
- 不新增独立“防复发中心”页面。

## 9. 可访问性与视觉约束

- 以 `1440x900` 为主要验证尺寸；
- 表格保持固定高度滚动，不产生页面横向溢出；
- 抽屉和弹窗不得遮挡自身底部操作区；
- 行、按钮和状态变化均可通过键盘操作；
- 抽屉和弹窗支持 Escape 关闭；
- 弹窗需要焦点圈定，关闭后焦点返回触发行；
- 状态不能只依赖颜色表达；
- 不改变公共 Sidebar、Topbar 和其他分析卡片布局。

## 10. 首版范围

首版必须实现：

- 稳定 `issueId` 和问题指纹；
- 表格治理状态与行详情交互；
- 防复发详情抽屉；
- 创建关联治理任务；
- 活动任务去重；
- 状态机与审计记录；
- 任务完成后进入观察期；
- 确认稳定、重新治理和回滚；
- 治理前后效果与副作用指标；
- 时间范围切换时保留治理状态。

明确延后：

- AI 自动确认根因；
- 自动修改库存、物流或平台规则；
- 自动拆分与合并问题簇；
- 多层级可配置审批流；
- 跨店铺治理知识库；
- 严格因果归因和自动灰度止损；
- 新建独立防复发页面。

## 11. 验收标准

1. 点击或键盘操作表格行可打开正确的问题详情。
2. 抽屉展示完整建议、复发证据、根因、方案、关联任务和审计记录。
3. 创建治理任务后，任务协同只出现一条与该 `issueId` 关联的任务。
4. 重复点击、重复提交和切换时间范围均不能生成第二条活动任务。
5. 创建失败时表单数据保留，任务和问题状态均不发生部分更新。
6. 治理任务完成后问题进入观察中，不直接进入已稳定。
7. 观察达到阈值并人工确认后才能进入已稳定。
8. 已稳定问题再次超过阈值时重新打开原治理记录。
9. 忽略、审批、驳回、回滚和重新治理均写入审计记录。
10. 时间范围切换只改变统计值，不改变问题状态或任务关联。
11. `1440x900` 下表格、抽屉和弹窗无横向溢出、遮挡或布局跳动。
12. 抽屉和弹窗支持 Escape、键盘导航、焦点圈定和焦点恢复。

## 12. 反目标

以下结果不视为完成防复发治理：

- 只展示更长的 AI 建议；
- 只增加详情抽屉；
- 点击后只生成普通任务；
- 任务完成即标记问题解决；
- 仅用原始出现次数排序；
- 没有基线和口径却宣称挽回金额；
- 自动执行高风险措施但没有审批、审计和回滚。
