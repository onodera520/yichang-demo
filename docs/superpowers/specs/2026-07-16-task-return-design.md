# 任务退回与重新打开设计

## 目标

为任务协同页补齐可审计、可定位的退回能力。退回不删除处理记录，也不执行任意历史撤销；它表示将任务送回一个明确的业务环节重新处理。

## 交互定义

详情区根据当前状态展示不同动作：

- `待确认`、`已升级`：显示“退回”，退回到 `处理中`，负责人保持不变。
- `已分派`：显示“撤销分派”，退回到 `待分派`，负责人改为 `未分派`。
- `处理中`、`待分派`、`已超时`：不显示退回动作。处理中任务使用“转交”，待分派任务先分派，已超时任务保留当前处理动作。
- `已完成`：显示“重新打开”，恢复到 `处理中`；若负责人无效则恢复到 `待分派`。

点击“退回”“撤销分派”或“重新打开”后弹出确认框。退回原因必填，最多 200 字。确认框展示当前状态和目标状态，避免用户误解为删除最新日志。

## 状态与数据

新增纯状态转换函数，根据任务当前状态返回目标状态和字段补丁。无合法转换时不修改任务。

所有动作均追加处理记录：

- action：`退回任务`、`撤销分派` 或 `重新打开任务`
- detail：包含退回原因和 `原状态 → 目标状态`
- owner：当前操作人；Demo 中使用当前负责人，无负责人时使用 `系统`
- tone：退回使用蓝色，重新打开使用橙色

处理记录只追加，不删除、不覆盖。连续退回不会丢失此前的完成、升级或转交记录。

重新打开已完成任务时：

- 清除完成凭证展示所依赖的 `completionEvidence`。
- 恢复 SLA。完成动作先把原 SLA 写入 `previousRemainingSLA`；重新打开时优先恢复该值，旧任务没有该字段时使用 `04:00:00`。
- 来源为订单时，将已同步为 `已完成` 的来源订单恢复为 `处理中`。
- 来源为库存时，仅撤销由完成动作写入的 `已完成` 状态；不反向推断或恢复风险等级，避免覆盖后续库存变化。
- 若来源对象不存在，阻止重新打开并显示错误 Toast。

## Tab 提醒

退回后复用现有 `taskTabNotices`：

- 退回到 `处理中` 时，“处理中”Tab 增加对应任务提醒。
- 撤销分派后，“待分派”Tab 增加对应任务提醒。
- 点击目标 Tab 后继续沿用清除筛选、置顶、打开详情和三秒高亮规则。
- 同一任务连续流转时，提醒只保留在最新目标 Tab 中。

## 组件与接口

新增可复用 `TaskReturnDialog`，接收任务、动作类型、目标状态、确认和关闭回调。弹窗负责原因校验，不直接修改任务状态。

在状态层新增：

```js
returnTask(taskId, { reason })
reopenTask(taskId, { reason })
```

状态层统一更新任务、来源对象和 Tab 提醒，页面只负责打开弹窗和展示 Toast。

## 错误处理

- 原因为空：禁用确认按钮并显示字段提示。
- 状态已变化或不支持退回：不更新数据，Toast 提示“当前状态无法退回”。
- 重新打开时来源对象缺失：不更新数据，Toast 提示“来源对象不存在，无法重新打开”。
- 重复点击确认：提交期间禁用按钮，保证一次操作只追加一条日志。

## 测试

- 纯状态转换：覆盖每种状态的合法目标和非法状态不变。
- 处理记录：验证日志追加、原因和状态路径完整，旧日志不被删除。
- 来源同步：完成任务重新打开后订单恢复处理中；库存状态撤销但风险等级不被反向覆盖。
- 负责人规则：撤销分派清空负责人；重新打开无负责人任务进入待分派。
- Tab 提醒：退回目标 Tab 计数、迁移、去重及点击定位。
- 页面交互：原因必填、取消不修改、重复提交防护、Toast 和按钮文案随状态变化。
- 运行 `node --test`、`npm.cmd run build`，并在 1440x900 下验证弹窗、按钮和目标行高亮。

## 非目标

- 不支持任意选择历史状态。
- 不删除或修改已有处理记录。
- 不实现多级审批、权限系统或后端持久化。
- 不自动恢复库存风险等级或其他可能已被后续流程修改的数据。
