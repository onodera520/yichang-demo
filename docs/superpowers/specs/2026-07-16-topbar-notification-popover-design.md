# 顶部消息通知悬浮窗口设计

## 目标

让顶部铃铛成为全局可用的消息快捷入口，同时继续保留异常工作台的“系统消息”作为完整消息中心。两处使用同一消息数据和已读状态，避免内容重复维护或未读数量不同步。

## 产品边界

- 顶部铃铛负责快速查看最新消息。
- 异常工作台负责查看、筛选和处理完整消息列表。
- 本次不新增独立通知页面、浏览器推送、声音提醒或跨会话持久化。
- 消息数据继续来自前端 mock 数据，已读状态继续保存在当前 React 会话中。

## 顶部铃铛

- 红色徽标显示当前可见消息中的真实未读数量，不再写死数字。
- 未读数超过 99 时显示 `99+`。
- 未读数为 0 时隐藏徽标。
- 点击铃铛切换悬浮窗口。
- 再次点击铃铛、点击窗口外区域或按 `Esc` 关闭窗口。
- 打开窗口本身不会把消息标记为已读。

## 消息悬浮窗口

- 定位在铃铛下方并与右侧边缘对齐，宽度约 380px。
- 标题为“消息通知”，右侧提供“全部已读”操作。
- 展示当前消息源中最新 5 条消息。
- 每条消息展示未读蓝点、标题、分类、时间和简短详情。
- 未读消息使用浅蓝背景，已读消息使用白色背景。
- 点击消息主体只展开或收起详情，并将该消息标记为已读，不发生页面跳转。
- 有业务目标的消息显示独立的“去处理”按钮。
- 只有点击“去处理”按钮才跳转到对应订单、库存、任务、设置或复盘位置，同时将该消息标记为已读。
- 没有业务目标的消息不显示跳转按钮。
- 底部提供“查看全部消息”，进入 `/dashboard` 并自动打开现有“系统消息”抽屉。
- 没有消息时展示空状态；没有未读消息时“全部已读”按钮禁用。

## 与异常工作台的关系

- 顶部悬浮窗口和异常工作台“系统消息”读取同一组消息。
- 两处共享 `readMessageIds`、`markMessageRead` 和 `markAllMessagesRead`。
- 任一入口标记已读后，另一入口的未读样式和数量立即同步。
- 异常工作台首页卡片仍展示最新消息摘要，不重复增加新的消息区域。
- 从顶部点击“查看全部消息”时，通过路由 state 请求 Dashboard 自动打开消息抽屉。
- Dashboard 消费该路由 state 后清理一次性打开意图，避免返回页面时反复自动打开。

## 组件与数据流

### NotificationPopover

新增可复用悬浮组件，负责：

- 渲染最新 5 条消息。
- 管理当前展开消息。
- 处理窗口内的消息已读、全部已读和显式跳转操作。
- 不自行复制或持久化消息数据。

### Topbar

Topbar 负责：

- 计算悬浮窗口开关状态。
- 获取共享消息、平台连接状态和已读状态。
- 渲染真实未读徽标。
- 将跳转目标交给 React Router。
- 处理点击外部和键盘关闭。

### 共享消息选择器

把当前 Dashboard 中基于平台连接状态过滤消息的规则提取为共享纯函数。Topbar 和 Dashboard 使用同一选择器，确保例如 eBay 连接恢复后，两处同时隐藏对应的过期平台消息。

## 可访问性

- 铃铛按钮提供包含未读数量的 `aria-label`。
- 悬浮窗口使用合适的弹出区域语义，并保持键盘可聚焦。
- `Esc` 可关闭窗口，关闭后焦点返回铃铛。
- “去处理”是明确的独立按钮，不让整条消息成为隐式导航入口。
- 未读状态除了颜色外，同时使用蓝点和可读文本语义表达。

## 测试范围

- 未读徽标使用真实数量，支持隐藏和 `99+`。
- 打开悬浮窗口不会自动清除未读。
- 点击消息主体只标记已读和展开详情，不跳转。
- 点击“去处理”才执行目标路由跳转。
- “全部已读”同步更新顶部和 Dashboard。
- “查看全部消息”进入 Dashboard 并自动打开消息抽屉。
- 点击外部、再次点击铃铛和按 `Esc` 均可关闭。
- 平台消息过滤规则在顶部和 Dashboard 保持一致。
- 在 1440x900 下窗口不超出视口、不遮挡头像区域，且无控制台错误。
